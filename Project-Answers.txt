5. Technical Decisions and Trade-offs
- Framework Choice: I chose Spring Boot with Java 17. Spring Boot's dependency injection and rich ecosystem (Spring Security, Spring Data JPA) make scaffolding standard CRUD applications and enforcing role-based access control significantly faster and robust.
- Database: I utilized an H2 in-memory database to keep the execution process extremely simple for evaluators. This eliminates the need to configure a local MySQL or PostgreSQL instance, ensuring zero-configuration deployment for review purposes. 
- API Design: The REST APIs are structured around clear principles, separating standard record management (`/api/records`) from analytical aggregation (`/api/dashboard`).
- Access Control: I implemented Spring Security with Basic Authentication instead of JWT for simplicity. This cleanly demonstrates Role-Based Access Control (@EnableMethodSecurity and SecurityFilterChain) mapped to user entries in the persistence layer, which fully addresses the requirements.
- Error Handling: A @RestControllerAdvice acts as a global exception handler. It formats backend validation errors (from jakarta.validation) into a clean, unified JSON structure instead of raw server stack traces.
